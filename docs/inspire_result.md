# 产品思路拓展深度研究结果

> 基于 `docs/inspire.md` 的5个方向，结合参考项目（chat-export、ctxport、gemini-voyager）和 demo 现状的完整产品方案。

---

## 方向一：复杂内容抓取方案

### 内容类型矩阵

| 内容类型 | 能抓 | 技术难度 | 用户价值 | 存储成本 | 推荐阶段 |
|----------|------|----------|----------|----------|----------|
| 纯文本 | ✅ | 低 | 核心 | 低 | MVP（已有） |
| 代码块 | ✅ | 低 | **极高(60-80%)** | 低 | MVP（已有，需处理CodeMirror `<br>` 结构） |
| 富文本格式 | ✅ | 低 | 中高(20-30%) | 无 | MVP(API)/P2(DOM) |
| 网页链接 | ✅ | 低 | 中 | 无 | MVP（AI总结时被动吸收） |
| 表格 | ✅ | 低 | 中(10-15%) | 低 | MVP(API)/P2(DOM) |
| 图片URL | ✅ | 中 | 低(5-10%) | 低(存URL) | **Phase 2（只存URL，不转base64）** |
| 公式(LaTeX) | ✅ | 中 | 低-中(5-10%) | 低 | Phase 2 |
| 图片base64 | ⚠️ | 高 | 低 | **极高** | **不建议做** |
| 文件附件内容 | ❌ | 极高 | 极低 | 极高 | **可以永远不做** |

### 关键产品决策

1. **MVP坚持"纯文本优先"**：知识沉淀的核心价值在文本内容（见解、产出、代码），不是图片
2. **图片永远存URL，不存base64**：base64会让SQLite膨胀10-100倍，展示时尝试加载URL，失败显示"图片已失效"
3. **文件附件内容永远不需要单独抓**：AI模型已经解析了上传文件并将关键内容体现在回复中
4. **内部API vs DOM路径差异**：内部API下表格/富文本已是结构化数据（零代价），DOM路径需逐一适配提取器

### 参考项目技术复用

| 参考文件 | 能力 | 代码量 |
|----------|------|--------|
| `chat-export/src/modules/content-handlers.ts` | `extractFormattedText()` DOM→Markdown递归转换，处理`<strong>/<em>/<code>/<h1-h6>/<ul>/<ol>/<a>` | ~60行 |
| `ctxport/packages/core-plugins/src/plugins/chatgpt/content-flatteners/` | ContentFlattener注册表模式，处理text/code/thoughts/multimodal/tool_response/reasoning_recap等8种类型 | 策略模式 |
| `gemini-voyager/src/features/export/services/DOMContentExtractor.ts` | 最完整的DOM提取器，支持math-block/math-inline(LaTeX)、table-block、Shadow DOM穿透 | ~200行 |

### 桌面客户端抓取结论

| 方案 | 可行性 | 建议 |
|------|--------|------|
| Accessibility API | 技术可行但代价极高 | Phase 3再考虑 |
| 屏幕截图+OCR | 准确率低(~85%)，代码/公式识别极差 | 不做 |
| 读取本地SQLite | 需逆向数据库schema，macOS沙箱限制 | Phase 3调研Claude Desktop |
| **当前建议** | **不做，引导用户用网页版** | ROI为负 |

## 方向二：意图识别与智能总结

### 分类体系：7大类 + 26子类

| 大类 | 占比预估 | 子类 | 特征信号 |
|------|---------|------|---------|
| **A 知识探讨** | 25% | A1概念解释/A2原理深挖/A3理论框架/A4对比辨析/A5学习路径 | "是什么"/"为什么"/"原理"/"机制"/多轮追问/概念密度高 |
| **B 操作指南** | 20% | B1功能使用/B2配置部署/B3报错排查/B4最佳实践/B5工具推荐 | "怎么"/"如何"/"步骤"/"配置"/步骤式回答/代码+参数 |
| **C 创作产出** | 20% | C1文案文章/C2代码生成/C3方案计划/C4翻译改写/C5迭代修改 | 长创造性输出/有修改迭代/用户给反馈让改 |
| **D 事实查询** | 15% | D1定义术语/D2数据参数/D3时间事件/D4推荐清单 | 1-2轮/直接答案/无深入讨论/事实性 |
| **E 分析决策** | 10% | E1方案评估/E2数据解读/E3利弊权衡/E4趋势预判 | "选哪个"/"哪个好"/对比表格/决策依据 |
| **F 头脑风暴** | 5% | F1创意发散/F2场景推演/F3开放讨论/F4自我反思 | 开放式问题/多个方向/无唯一答案 |
| **G 其他** | 5% | G1闲聊/G2多意图混合/G3无法识别 | 无明确目标/话题跳跃 |

### 多意图混合对话拆分策略

不是简单合并或拆分，而是按**话题切换点**切割：
- 连续 3+ 轮同一子话题 = 一个话题块
- 话题块 >= 2 且至少一个 >= 5 轮 = **拆分为多张卡片**
- 过短的子话题（1-2轮）合并到相邻主话题

### 不同意图的总结策略

| 类型 | insights 侧重 | outputs 侧重 | 压缩程度 |
|------|-------------|-------------|---------|
| 知识探讨 | 概念定义、认知突破、遗留疑问 | 概念框架 | 中 |
| 操作指南 | 注意事项、常见坑 | **可执行步骤清单** | 高 |
| 创作产出 | 创作思路、关键决策 | **最终产出全文（不压缩）** | 低 |
| 事实查询 | 无 | 1-2行答案 | 极高 |
| 头脑风暴 | 探索方向清单 | 创意草稿 | 高 |

### 实现路径：渐进式

| 阶段 | 方案 | API成本（单次） |
|------|------|-----------------|
| MVP | **单Prompt（条件分支）** — 一个system prompt处理所有类型 | ~¥0.005 |
| Phase 2 | 前置轻量分类器 + 路由到专用Prompt | ~¥0.007 |
| Phase 3 | 本地规则分类器 + 完整Multi-Agent | ~¥0.005 |

**结论：成本差异不到1分钱/次，Multi-Agent的准确性和可维护性优势完全值得在Phase 2引入。**

### 数据模型扩展（建议新增字段）

| 字段 | 类型 | 说明 |
|------|------|------|
| `card_type` | string | 意图大类（knowledge/operation/creation/fact/analysis/brainstorm/other） |
| `intent_code` | string | 26子类代码（A1-A5, B1-B5, ...） |
| `unresolved_questions[]` | string[] | 遗留问题（知识探讨型特有） |
| `exploration_paths[]` | string[] | 探索方向（头脑风暴型特有） |
| `draft_of` | string | 草稿关联终稿的card_id（创作产出型特有） |
| `summary_confidence` | number | 总结置信度（0-1） |

### 核心Prompt模板（MVP可直接使用）

#### 知识探讨型

```
你是一个对话分析专家。请分析以下用户与AI的对话，生成结构化的知识卡片。

这段对话属于"知识探讨"类型——用户在与AI深入讨论某个概念、理论或知识点。

请按以下JSON格式输出：
{
  "title": "对话主题（5-10字）",
  "original_question": "用户的核心疑惑（1-2句）",
  "key_concepts": ["核心概念1", "核心概念2"],
  "insights": ["关键认知突破/理解收获", ...最多5条],
  "unresolved_questions": ["对话中尚未解决的问题/可能的后续探索方向", ...最多3条],
  "tags": ["3-5个标签"],
  "topic": "从以下类别中选择或自定义：知识科普/技术原理/学习笔记/理论探讨/其他"
}

约束：
- 如果对话涉及多个相关概念，请提炼它们之间的关系
- insights要写"用户获得了什么新理解"，不是"对话说了什么"
- unresolved_questions要捕捉对话中暗示但未深入的方向
- 输出纯JSON，不要其他内容
```

#### 操作指南型

```
你是一个对话分析专家。请分析以下用户与AI的对话，生成结构化的知识卡片。

这段对话属于"操作指南"类型——用户在询问如何实现某个功能、配置或解决报错。

请按以下JSON格式输出：
{
  "title": "操作目标（5-10字）",
  "original_question": "用户想做什么/遇到了什么报错",
  "steps": ["可执行的步骤1", "步骤2", ...],
  "insights": ["注意事项/常见坑/关键参数", ...最多5条],
  "outputs": ["可复用的代码片段/配置模板/命令", ...最多3条],
  "tags": ["3-5个标签"],
  "topic": "从以下类别中选择或自定义：编程开发/配置部署/报错排查/工具使用/其他"
}

约束：
- steps要可直接执行，不要省略关键参数
- 如果对话中有多个方案，只保留最优方案
- 标注关键参数（如"--concurrency=10"）到insights中
- 输出纯JSON，不要其他内容
```

#### 创作产出型

```
你是一个对话分析专家。请分析以下用户与AI的对话，生成结构化的知识卡片。

这段对话属于"创作产出"类型——用户在让AI生成文案、代码、方案等创造性内容。

请按以下JSON格式输出：
{
  "title": "产出主题（5-10字）",
  "original_question": "用户要求AI做什么",
  "creation_type": "draft | final",
  "insights": ["创作思路/关键决策/修改原因", ...最多5条],
  "outputs": ["最终产出内容（保留完整，不压缩）", ...最多3条],
  "draft_of": "如果是草稿/修改过程，填终稿card_id；否则为空",
  "tags": ["3-5个标签"],
  "topic": "从以下类别中选择或自定义：文案写作/代码生成/方案计划/翻译改写/其他"
}

约束：
- 判断是否有修改迭代：如果有"改一下"/"换个风格"/"缩短一点"等反馈，creation_type="draft"
- 终稿要保留完整内容，不要压缩
- 如果是迭代过程，insights中记录每次修改的关键决策
- 输出纯JSON，不要其他内容
```

#### 事实查询型

```
你是一个对话分析专家。请分析以下用户与AI的对话，生成结构化的知识卡片。

这段对话属于"事实查询"类型——用户在问一个具体的事实性问题。

请按以下JSON格式输出：
{
  "title": "事实性答案（5-10字）",
  "original_question": "用户问了什么",
  "insights": ["不超过2条"],
  "outputs": ["1-2行直接答案"],
  "tags": ["2-3个标签"],
  "topic": "知识科普/其他"
}

约束：
- 极简压缩，控制在200字以内
- 如果对话仅1轮且答案极短（如"是"、"余华"），可不生成完整卡片
- 输出纯JSON，不要其他内容
```

## 方向三：知识卡片展示界面设计

### 5条设计原则

| 编号 | 原则 | 一句话解释 | 违反案例 |
|------|------|-----------|----------|
| P1 | **找得到 > 好看** | 搜索和筛选的可达性必须高于一切视觉装饰 | 首页放统计图但搜索框藏角落 |
| P2 | **密度随深度递增** | 列表页极简，详情页丰富，不在列表页堆信息 | 卡片列表展示完整见解+产出+对话 |
| P3 | **AI生成，人工策展** | 所有AI生成的内容必须可手动编辑和纠正 | 标签不可修改、总结不可调整 |
| P4 | **一个主视图** | 坚持"列表+筛选"极简模式，不做多视图切换 | 引入Board/Calendar/Timeline视图 |
| P5 | **正向激励** | 统计和反馈只展示正向数据，不惩罚、不焦虑 | "你已经30天没沉淀对话了" |

### 参考产品借鉴

| 参考产品 | 采纳 | 理由 |
|----------|------|------|
| flomo 时间线流式 | 部分 | 适合作为"最近沉淀"第二视图，不做主视图 |
| flomo #tag内联 | **采纳** | 标签系统最成功的设计，用户已认可 |
| flomo 每日回顾 | **采纳** | 核心黏性功能，可差异化实现 |
| Notion 多视图 | 不采纳MVP | 个人知识工具不需要Board/Calendar/Timeline |
| Notion [[双向链接]] | Phase 2 | 卡片量>100后才有意义 |
| Obsidian 知识图谱 | Phase 2（简化版） | 做"相似卡片推荐"列表，不做完整图谱 |
| Obsidian #parent/child层级标签 | **采纳** | 扁平标签在卡片量增长后必崩 |

## 方向四：用户管理能力

### 标签管理

| 决策 | 选择 | 理由 |
|------|------|------|
| 标签体系 | **层级标签**（`parent/child`） | 卡片量增长后扁平标签必崩 |
| 生成策略 | AI生成 + 用户可删除/补充 | 全自动质量不可控，纯手动用户懒得做 |
| 自动补全 | 输入`#`时弹出已有标签建议 | 降低输入成本 |
| 标签合并 | 设置页提供"合并标签"功能 | 避免`Python`和`python`并存 |
| 标签云 | **不做** | 视觉噪音大，信息密度低 |

### 卡片操作

| 操作 | 触发方式 | 阶段 |
|------|----------|------|
| 修改主题/标签 | 详情页内联编辑 | MVP |
| 归档 | 右键菜单 → "归档"（列表默认不显示） | P1 |
| 收藏 | 卡片上⭐图标 | P1 |
| 合并卡片 | 列表页多选 → "合并" → AI重新总结 | Phase 2 |
| 拆分卡片 | 详情页选择部分对话 → "拆分为新卡片" | Phase 2 |

### 分享卡片

- **分享图片**：标题 + 核心问题 + 精选见解 + 来源 + 产品水印
- **导出Markdown**：完整卡片内容（已有）
- **隐私红线**：原始对话**绝对不能**暴露，平台URL不暴露

### 统计面板（有价值的 vs 噱头）

| 统计项 | 用户价值 | 黏性贡献 | 推荐 |
|--------|----------|----------|------|
| 本月沉淀对话数 | 量化知识积累成就感 | 高 | ✅ |
| **连续使用天数(streak)** | 游戏化，激励持续使用 | **极高**（参考Duolingo/GitHub） | ✅ **ROI最高** |
| 最常讨论话题Top 5 | 帮助了解自己关注点 | 中 | ✅ |
| 知识增长曲线（月累计） | 长期回顾的情感价值 | 中 | P1 |
| 平台使用分布 | 了解工具偏好 | 低 | 可做 |
| "你的知识超过X%用户" | 个人工具无社交比较需求 | 无 | ❌ |
| "今日知识值+10" | 虚拟货币无意义 | 无 | ❌ |

## 方向五：知识库复习激活机制

### 复习内容生成策略（按卡片类型智能路由）

| 卡片类型 | 复习素材 | 示例 | 生成成本 |
|----------|----------|------|---------|
| 事实/知识型 | **陈述性解读** | "你曾在3月问过活着的作者是谁，答案是余华" | 低（总结prompt追加1条指令） |
| 原理/概念型 | **关键见解提取** | "Python的GIL导致多线程无法真正并行" | 极低（取insights[0]） |
| 方法/方案型 | **问题+答案摘要** | "关于Python并行，核心要点是用multiprocessing代替threading" | 低 |
| 代码/技术型 | **问题+方案摘要** | "你解决过多爬虫并发问题，关键方案是Semaphore" | 低 |

**生成时机**：在AI总结prompt末尾追加指令，不增加额外API调用。

### 复习时机算法：艾宾浩斯骨架 + 场景触发皮肤

```
每张卡片的复习状态：
{
  review_schedule: [1, 2, 4, 7, 15, 30],  // 艾宾浩斯间隔（天）
  review_history: [{ date, interval, acknowledged }],
  mastered: false,
  next_review_date: "2026-05-26"
}
```

**触发条件（满足任一）：**
1. 用户打开桌面应用时 → 检查待复习卡片
2. 后台定时（每小时）→ 待复习卡片 && 应用在前台 → 系统通知
3. 新卡片入库后达到3张未复习 → 触发"新卡片回顾"

**每日上限**：最多5张（防止轰炸）

**卡片选择优先级：** 超期最久 > 用户搜索过的标签 > insights数量多 > 最近创建

### 交互流程

```
[系统通知]  📖 今日回顾：余华是《活着》的作者
        ↓ 点击展开
┌─────────────────────────────────────────────┐
│ 📅 2025年3月你曾问过                         │
│ "《活着》的作者是余华吗？"                    │
│                                             │
│ ── 答案 ──                                  │
│ 是的。余华（1960-）是中国当代作家...          │
│                                             │
│ [查看完整卡片]  [我记住了 ✓]  [稍后再说]      │
└─────────────────────────────────────────────┘
```

### 不同卡片类型的复习策略

| 卡片类型 | 复习价值 | 处理方式 |
|---------|---------|---------|
| 事实/知识型 | ✅ 高价值 | 正常艾宾浩斯间隔 |
| 原理/概念型 | ✅ 高价值 | 正常艾宾浩斯间隔 |
| 方案/决策型 | ⚠️ 中价值 | 降低频率，30天一次 |
| 操作指南型 | ❌ 低价值 | 不主动推送，搜索时自然出现 |
| 代码片段型 | ⚠️ 中价值 | 生成"你解决过XX问题"的素材 |

### 用户控制（三级粒度）

1. **总开关**（设置页）：开启/关闭所有复习提醒
2. **卡片级别**（详情页）：该卡片是否参与复习
3. **频率控制**（设置页）：每日推送上限（3/5/10）
