# PRD v2：LLM 对话自动沉淀知识库工具

> 更新时间：2026-05-22
> 版本：v2（融合 PRD-v1 + 5方向深度产品方案，不含版本迭代概念，按最高版本采纳）

---

## 1. 产品定位

**一句话：** 把你和大模型的对话，自动变成可检索、可复习的个人知识库。

**核心痛点：** 用户日常与 ChatGPT、豆包、DeepSeek 等大模型对话，产出大量知识、方案、代码和决策依据，但几乎不会手动整理；各平台历史记录杂乱、不可跨平台检索，有价值的内容持续沉没。

**核心价值：** 一键（或自动）将散落在各平台的对话提炼为结构化知识卡片，归入本地私有知识库，随时可检索、可复盘、可导出、可复习。

---

## 2. 目标用户

| 维度 | 描述 |
|------|------|
| 画像 | 高频使用 2+ 个网页版 LLM 的知识工作者 |
| 典型场景 | 程序员用 GPT 写代码、产品经理用豆包做方案、研究者用 DeepSeek 查文献 |
| 共性需求 | 把对话变笔记/方案库/代码片段库，且不想手动复制整理 |
| 隐私倾向 | 偏好本地存储，不想把对话再上传到第三方 |

## 3. 核心使用场景

### 场景 A：一键沉淀

```
用户在 LLM 页面聊完一个话题
        ↓
点击浏览器页面的悬浮球
        ↓
扩展自动抓取完整对话（内部接口优先）
        ↓
发送到桌面应用后台
        ↓
AI 识别意图类型 → 路由到对应 Prompt → 生成知识卡片
        ↓
归入本地知识库
        ↓
用户在桌面应用中搜索、浏览、导出、复习
```

### 场景 B：智能提醒

```
用户在 LLM 页面进行了一段对话后离开
        ↓
扩展检测到对话空闲 > N 分钟
        ↓
悬浮球变为提示状态（"有新对话可沉淀"）
        ↓
用户一键确认 → 执行抓取+总结+入库
```

### 场景 C：自动沉淀

```
扩展后台持续监听 LLM 页面活动
        ↓
按时间窗口自动分段（如 30 分钟无活动视为一段）
        ↓
自动抓取 + 意图识别 + 总结 + 入库
        ↓
仅在通知栏显示结果
```

> **"对话结束"没有明确的技术信号，是核心设计难点。** 分阶段实现，从手动到自动，让用户始终有控制感。

### 场景 D：知识复习

```
用户打开桌面应用（或系统通知触发）
        ↓
回顾最近值得重温的知识卡片
        ↓
弹窗展示复习内容（按卡片类型智能生成）
        ↓
用户标记"我记住了" → 更新复习进度
        ↓
完成今日回顾，或达到每日上限
```

### 4.2 为什么不用纯浏览器扩展？

| 问题 | 说明 |
|------|------|
| 知识库界面空间不足 | Side Panel/Popup 只有窄条，无法做复杂的浏览、筛选、详情阅读 |
| 无独立窗口 | 用户想"打开知识库认真翻一翻"时，必须先开浏览器再开侧边栏 |
| 跨浏览器数据隔离 | Chrome 和 Edge 数据不互通 |
| 没有全局快捷键 | 无法做到系统级随时唤出 |
| 存储不可靠 | IndexedDB 可能被浏览器清理 |
| 无法后台运行 | 浏览器关了就没有自动发现能力 |

### 4.3 为什么选 Tauri 而非 Electron？

| 对比 | Tauri 2.0 | Electron |
|------|-----------|----------|
| 包体积 | ~10MB | ~150MB |
| 内存占用 | 低 | 高 |
| 跨平台 | ✅ 同一份代码构建 Windows (.msi) + Mac (.dmg) | ✅ 同 |
| 平台差异处理 | Tauri API 自动抹平（路径、快捷键、托盘等） | 需自行处理 |
| 安全性 | Rust 后端，权限系统更严格 | Node.js 后端 |
| 移动端扩展 | Tauri 2.0 支持 iOS/Android | 不支持 |

### 4.4 跨平台策略

- 前端（React + HTML/CSS）：100% 平台无关
- 后端（Tauri Rust API 调用）：Tauri 自动处理平台差异
- 构建：`tauri build` 指定 target 即可产出 .msi（Windows）或 .dmg（Mac）
- 唯一需注意的 ~1% 差异：
  - 快捷键用 `CmdOrCtrl+K`（Tauri 自动映射）
  - 数据路径用 `appDataDir()` API（自动返回正确路径）
  - 窗口按钮位置 CSS 适配

## 5. 技术架构

### 5.1 浏览器扩展（抓取探头）

**职责单一：只负责识别平台 + 抓取对话 + 传给桌面应用。**

| 组件 | 职责 |
|------|------|
| Content Script | 注入悬浮球、识别当前平台 |
| Platform Provider | 按平台实现抓取逻辑（内部接口 / DOM / Shadow DOM） |
| Background Service Worker | 通过 HTTP POST 将抓取结果发送到桌面应用 |

### 5.2 平台 Provider 接口

```ts
interface ChatProvider {
  id: string;                                    // "doubao" | "chatgpt" | "deepseek" | ...
  match(url: string): boolean;                   // 匹配当前页面
  extractCurrent(): Promise<RawConversation>;    // 抓取当前对话
  fetchById?(id: string): Promise<RawConversation>; // 按 ID 拉取历史对话
  injectButton?(): void;                         // 注入悬浮球
}

interface RawConversation {
  platform: string;
  conversationId: string;
  title?: string;
  url: string;
  messages: Array<{
    role: "user" | "assistant";
    content: string;
    timestamp?: string;
  }>;
  capturedAt: string; // ISO datetime
}
```

### 5.3 抓取优先级

1. **平台内部接口读取**（最稳）— 如豆包 `/im/chain/single`
2. **平台稳定语义选择器** — 如 ChatGPT `data-testid` 属性
3. **通用 DOM / open Shadow DOM 深遍历** — 兜底方案
4. **复制按钮、编辑态、分享页** — UI 旁路
5. **OCR / 截图** — 极低优先级兜底

### 5.4 桌面应用架构

| 层 | 技术 | 职责 |
|----|------|------|
| 前端 | React + TypeScript | 知识库 UI、设置页、复习弹窗、统计面板、导入导出 |
| 后端 | Tauri (Rust) | 本地 HTTP 服务、SQLite、API 调用代理、文件系统、系统托盘、复习调度 |
| 通信 | HTTP localhost:17321 | 扩展 → 桌面应用的本地通信 |

### 5.5 数据分层

```
Raw（原始抓取结果）→ Clean（清洗后纯文本问答）→ KnowledgeCard（AI 总结卡片）
```

## 6. 内容抓取策略

### 6.1 内容类型矩阵

| 内容类型 | 技术方案 | 阶段 |
|----------|---------|------|
| 纯文本 | 内部 API / DOM 提取 | 核心（默认） |
| 代码块 | CodeMirror `<br>` 结构解析 | 核心（需修复） |
| 富文本格式 | 内部 API（结构化数据） / DOM 递归转换 | 核心 |
| 网页链接 | AI 总结时被动吸收 | 核心 |
| 表格 | 内部 API（零代价） / DOM 提取器 | 核心 |
| 图片 | 只存 URL，不转 base64 | 核心 |
| 公式（LaTeX） | DOM 提取 `math-block` / `math-inline` | 核心 |
| 文件附件内容 | **永远不做** — AI 已在回复中解析关键内容 | 排除 |

### 6.2 关键产品决策

1. **纯文本优先**：知识沉淀的核心价值在文本内容（见解、产出、代码），不是图片
2. **图片只存 URL**：base64 会让 SQLite 膨胀 10-100 倍，展示时尝试加载 URL，失败显示"图片已失效"
3. **文件附件内容永远不需要单独抓**：AI 模型已经解析了上传文件并将关键内容体现在回复中
4. **内部 API vs DOM 路径差异**：内部 API 下表格/富文本已是结构化数据（零代价），DOM 路径需逐一适配提取器
5. **桌面客户端抓取不做**：ROI 为负，引导用户用网页版

### 6.3 参考项目代码复用

| 参考文件 | 能力 | 用途 |
|----------|------|------|
| `chat-export/src/modules/content-handlers.ts` | `extractFormattedText()` DOM→Markdown 递归转换 | 富文本提取 |
| `ctxport/packages/core-plugins/src/plugins/chatgpt/content-flatteners/` | ContentFlattener 注册表模式，处理 8 种内容类型 | 策略模式参考 |
| `gemini-voyager/src/features/export/services/DOMContentExtractor.ts` | 最完整的 DOM 提取器，支持 LaTeX/表格/Shadow DOM | DOM 提取器参考 |

## 7. AI 总结与意图识别

### 7.1 意图分类体系

10 个意图大类，覆盖用户与 LLM 对话的全部使用场景。每张知识卡片只归属于一个大类，不再划分子类：

| 大类 | 英文 key | 占比预估 | 特征信号 |
|------|----------|---------|---------|
| **概念理解** | `concept_exploration` | 20% | "是什么""为什么""原理是""机制""解释一下" |
| **事实查询** | `fact_lookup` | 15% | "谁""哪年""多少钱""是什么东西"，1-2 轮直接答案 |
| **技能学习** | `skill_learning` | 10% | "教我""学习计划""这道题""备考""怎么学" |
| **操作指南** | `how_to` | 15% | "怎么""如何配置""报错了""步骤" |
| **内容创作** | `content_creation` | 20% | "帮我写""生成一个""做个PPT""写代码" |
| **文本处理** | `text_processing` | 10% | "帮我改""润色""翻译""整理一下""提取""转成JSON" |
| **规划决策** | `planning_decision` | 5% | "该选哪个""分析一下""帮我规划""优劣""计划" |
| **头脑风暴** | `brainstorming` | 5% | "帮我想想""有什么创意""来10个""头脑风暴" |
| **交互陪伴** | `interactive_companion` | 3% | "扮演""模拟一下""练英语""心情不好""聊聊" |
| **其他** | `other` | 2% | "在吗""随便聊聊""无法识别" |

### 7.2 多意图对话拆分策略

- 连续 3+ 轮同一话题 = 一个话题块
- 话题块 ≥ 2 且至少一个 ≥ 5 轮 = **拆分为多张卡片**
- 过短的子话题（1-2 轮）合并到相邻主话题
- 每张卡片归属一个意图大类，不再划分子类

### 7.3 个性化主题生成

LLM 在总结时提炼一个 **简洁、概括性的名词或短句** 作为卡片的个性化主题，例如：
- `编程`、`产品设计`、`市场营销`、`Python并发`、`旅行攻略`
- 不是完整句子，而是领域/话题的标签，便于快速浏览和筛选

### 7.4 不同意图的总结策略

| 类型 | insights 侧重 | outputs 侧重 | 压缩程度 |
|------|-------------|-------------|---------|
| 概念理解 | 概念定义、认知突破、遗留疑问 | 概念框架 | 中 |
| 事实查询 | 无 | 1-2 行答案 | 极高 |
| 技能学习 | 学习路径、关键难点、易错点 | 练习题/笔记摘要 | 中 |
| 操作指南 | 注意事项、常见坑 | **可执行步骤清单** | 高 |
| 内容创作 | 创作思路、关键决策 | **最终产出全文（不压缩）** | 低 |
| 文本处理 | 修改说明、风格变化 | 处理后文本 | 低 |
| 规划决策 | 分析维度、权衡依据 | **方案对比 / 计划清单** | 中 |
| 头脑风暴 | 探索方向清单 | 创意草稿 | 高 |
| 交互陪伴 | 对话模式、用户反馈 | 无 | 极高 |

### 7.5 实现架构：单 Prompt 意图识别 + 主题提炼

采用单 Prompt 同时完成意图大类判定 + 个性化主题提炼 + 卡片总结。每个意图类型的专用 Prompt 模板独立维护在 `docs/prompts/` 目录下，不嵌入 PRD。

### 7.5 推荐模型与成本

| 模型 | 输入/1M tokens | 输出/1M tokens | 单次成本（~10K in + 1K out） | 推荐度 |
|------|---------------|---------------|---------------------------|--------|
| **GPT-4.1 nano** | $0.10 | $0.40 | ≈ ¥0.01 | ⭐⭐⭐ 首选 |
| **Gemini 2.5 Flash-Lite** | $0.10 | $0.40 | ≈ ¥0.01 | ⭐⭐⭐ 同级别 |
| **DeepSeek V3** | ¥2 | ¥8 | ≈ ¥0.03 | ⭐⭐ 国内首选 |

**月度成本估算（GPT-4.1 nano）：**
- 轻度（150次/月）：≈ ¥0.27
- 中度（450次/月）：≈ ¥0.81
- 重度（900次/月）：≈ ¥1.62

**结论：API 成本极低，不是商业模式的障碍。**

### 7.6 长对话优化策略

- **截取策略**：大多数对话关键信息集中在后半段，可只发送最近 N 轮
- **分片总结**：长对话分段总结再合并，两次小调用比一次大调用更便宜
- **渐进式总结**：追加问题时只总结新增部分，与已有卡片合并
- **缓存命中**：相同 system prompt 部分走 DeepSeek 缓存，输入成本降 75%

### 7.7 本地模型（可选，零成本）

- 用 Ollama 跑 Qwen2.5-7B / Llama3-8B
- 完全免费，但需用户有 8GB+ 显存
- 作为"离线模式"选项

## 8. 知识卡片展示界面

### 8.1 5 条设计原则

| 编号 | 原则 | 一句话解释 |
|------|------|-----------|
| P1 | **找得到 > 好看** | 搜索和筛选的可达性必须高于一切视觉装饰 |
| P2 | **密度随深度递增** | 列表页极简，详情页丰富，不在列表页堆信息 |
| P3 | **AI生成，人工策展** | 所有 AI 生成的内容必须可手动编辑和纠正 |
| P4 | **一个主视图** | 坚持"列表+筛选"极简模式，不做多视图切换 |
| P5 | **正向激励** | 统计和反馈只展示正向数据，不惩罚、不焦虑 |

## 9. 用户管理能力

### 9.1 标签管理

| 决策 | 选择 | 理由 |
|------|------|------|
| 标签体系 | **层级标签**（`parent/child`） | 卡片量增长后扁平标签必崩 |
| 生成策略 | AI 生成 + 用户可删除/补充 | 全自动质量不可控，纯手动用户懒得做 |
| 自动补全 | 输入 `#` 时弹出已有标签建议 | 降低输入成本 |
| 标签合并 | 设置页提供"合并标签"功能 | 避免 `Python` 和 `python` 并存 |

### 9.2 卡片操作

| 操作 | 触发方式 |
|------|----------|
| 修改标题/标签 | 详情页内联编辑 |
| 归档 | 右键菜单 → "归档"（列表默认不显示） |
| 收藏 | 卡片上 ⭐ 图标 |
| 合并卡片 | 列表页多选 → "合并" → AI 重新总结 |
| 拆分卡片 | 详情页选择部分对话 → "拆分为新卡片" |

### 9.3 分享功能

- **分享图片**：标题 + 核心问题 + 精选见解 + 来源 + 产品水印
- **导出 Markdown**：完整卡片内容
- **隐私红线**：原始对话绝对不能暴露，平台 URL 不暴露

### 9.4 统计面板

| 统计项 | 推荐 | 理由 |
|--------|------|------|
| 本月沉淀对话数 | ✅ | 量化知识积累成就感 |
| **连续使用天数（streak）** | ✅ | 游戏化，激励持续使用，ROI 最高 |
| 最常讨论话题 Top 5 | ✅ | 帮助了解自己关注点 |
| 知识增长曲线（月累计） | ✅ | 长期回顾的情感价值 |
| 平台使用分布 | 可做 | 了解工具偏好 |

### 9.5 Streak 设计

```
┌─────────────────────────────────┐
│ 🔥 连续 5 天                    │
│ 一 二 三 四 五 六 日             │
│ ● ● ● ● ● ○ ○                  │
│                                 │
│ 本月已沉淀 42 段对话             │

```

断掉时显示"你的连续记录断了，今天沉淀一段来续上？"——**只正向激励，不惩罚**。

## 10. 知识库复习激活机制

### 10.1 复习内容生成策略

按卡片类型智能路由生成复习素材：

| 卡片类型 | 复习素材 | 示例 | 生成成本 |
|----------|----------|------|---------|
| 事实/知识型 | 陈述性解读 | "你曾在3月问过活着的作者是谁，答案是余华" | 低 |
| 原理/概念型 | 关键见解提取 | "Python的GIL导致多线程无法真正并行" | 极低 |
| 方法/方案型 | 问题+答案摘要 | "关于Python并行，核心要点是用multiprocessing代替threading" | 低 |
| 代码/技术型 | 问题+方案摘要 | "你解决过多爬虫并发问题，关键方案是Semaphore" | 低 |

**生成时机**：在 AI 总结 prompt 末尾追加指令，不增加额外 API 调用。

### 10.2 复习时机算法

艾宾浩斯骨架 + 场景触发皮肤：

```
每张卡片的复习状态：
{
  review_schedule: {
    intervals: [1, 2, 4, 7, 15, 30],  // 艾宾浩斯间隔（天）
    review_history: [{ date, interval, acknowledged }],
    mastered: false,
    next_review_date: "2026-05-26"
  }
}
```

**触发条件（满足任一）：**
1. 用户打开桌面应用时 → 回顾最近值得重温的卡片
2. 后台定时（每小时）→ 有可回顾卡片 && 应用在前台 → 系统通知
3. 新卡片入库后达到 3 张未回顾 → 触发"新卡片回顾"

**每日上限**：最多 5 张（每次只展示少量，体验不打扰）

**卡片选择优先级：** 距离上次回顾时间最长 > 用户搜索过的标签 > insights 数量多 > 最近创建

### 10.3 不同卡片类型的复习策略

| 卡片类型 | 复习价值 | 处理方式 |
|---------|---------|---------|
| 事实/知识型 | ✅ 高价值 | 正常艾宾浩斯间隔 |
| 原理/概念型 | ✅ 高价值 | 正常艾宾浩斯间隔 |
| 方案/决策型 | ⚠️ 中价值 | 降低频率，30 天一次 |
| 操作指南型 | ❌ 低价值 | 不主动提醒，搜索时自然出现 |
| 代码片段型 | ⚠️ 中价值 | 生成"你解决过 XX 问题"的素材 |

### 10.4 用户控制（三级粒度）

1. **总开关**（设置页）：开启/关闭所有复习提醒
2. **卡片级别**（详情页）：该卡片是否参与复习
3. **频率控制**（设置页）：每日展示上限（3/5/10）

### 10.5 交互流程

```
[系统通知]  📖 今日回顾：余华是《活着》的作者
        ↓ 点击展开
┌─────────────────────────────────────────────┐
│ 📅 2025年3月你曾问过                         │
│ "《活着》的作者是余华吗？"                    │
│                                             │
│ ── 答案 ──                                  │
│ 是的。余华（1960-）是中国当代作家...          │
│                                             │
│ [查看完整卡片]  [我记住了 ✓]  [稍后再说]      │
└─────────────────────────────────────────────┘
```

## 11. 排除项（明确不做）

| 排除项 | 原因 |
|--------|------|
| 图片 base64 存储 | SQLite 膨胀 10-100 倍，永远只存 URL |
| 文件附件内容抓取 | AI 已在回复中解析关键内容 |
| 桌面客户端（Claude Desktop 等）抓取 | ROI 为负，引导用户用网页版 |
| 屏幕截图 + OCR | 准确率 ~85%，代码/公式识别极差 |
| 标签云 | 视觉噪音大，信息密度低 |
| 虚拟货币统计（"知识值+10"） | 个人工具无意义 |
| 社交比较（"超过 X% 用户"） | 个人工具无社交比较需求 |
| 多视图切换（Board/Calendar/Timeline） | 个人知识工具不需要 |

## 13. 最大风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 平台内部接口变更/封禁 | 抓取失败 | 多层兜底策略（接口 → DOM → Shadow DOM），失败日志可观测 |
| "对话结束"判定模糊 | 自动场景体验差 | 分阶段：手动 → 提醒 → 自动，用户始终有控制权 |
| 用户不愿装两个东西（扩展+桌面应用） | 获客门槛高 | 安装引导要丝滑；扩展极轻量，桌面应用包体积小（~10MB） |
| AI 总结质量不稳定 | 知识卡片不好用 | 意图分类 + 专用 Prompt 路由提升准确性；用户可手动编辑纠正 |
| 跨平台 UI 一致性 | 维护成本 | Tauri 自动抹平；Web 渲染天然一致 |
| 复习提醒过于频繁 | 用户反感 | 每日上限 5 张 + 三级用户控制 + 正向激励 |
