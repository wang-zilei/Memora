# LLM 对话自动沉淀知识库工具：产品需求简报

## 1. 产品定位

开发一款轻量化的个人知识沉淀工具，自动抓取用户在各大网页版 LLM 产品中的完整对话，清洗后交给 AI 结构化总结，并归档为可检索、可复盘的私有知识库。

核心痛点是：用户日常和大模型对话能获得大量知识、方案、代码和决策依据，但后续通常懒得手动复制整理；平台历史记录又杂乱、不可跨平台检索，导致有价值内容沉没。

## 2. 目标用户

- 高频使用 ChatGPT、Claude、Gemini、豆包、Kimi、通义、DeepSeek 等网页版 LLM 的知识工作者
- 需要把对话沉淀为项目笔记、学习笔记、方案库、代码片段库的人
- 希望尽量本地化、私有化保存 AI 对话资产的用户

## 3. 核心使用流程

1. 用户打开某个平台的历史对话页。
2. 点击扩展按钮或页面内“一键沉淀”按钮。
3. 工具自动加载完整对话并抓取纯净问答内容。
4. 清洗器剔除导航、按钮、会话列表、来源卡片、思考过程等冗余内容。
5. 总结模型按固定模板生成结构化知识卡片。
6. 本地数据库保存原始对话、清洗后对话、总结结果、标签和来源 URL。
7. 用户在知识库页面搜索、筛选、复盘和导出。

## 4. MVP 范围

MVP 建议只做“豆包单平台 + 本地知识库 + 手动点击沉淀”：

- 浏览器扩展注入豆包页面
- 当前会话完整抓取
- 清洗为 `[{ role: "user" | "assistant", content: string }]`
- 调用总结模型生成：
  - 对话主题
  - 原始疑惑
  - 收获见解
  - 可复用落地产出
  - 自动标签
- 本地 IndexedDB 或 SQLite 保存
- 简单知识库页面：列表、搜索、详情、导出 Markdown

## 5. 非目标

- 不做绕登录、批量盗取、无授权抓取他人对话
- 不优先做全平台自动适配
- 不在 MVP 做团队协作、云同步、复杂权限系统
- 不依赖截图 OCR 作为主路径

## 6. 关键成功指标

- 豆包当前打开会话抓取成功率 >= 90%
- 长对话抓取完整率 >= 95%
- 清洗后导航/按钮/侧边栏垃圾文本明显少于 5%
- 用户点击一次后 30 秒内完成沉淀，超长对话允许异步处理
- 知识库内可按关键词搜到历史收获和落地产出

## 7. 最大技术风险

网页端 LLM 对话抓取是最大风险。风险包括 Shadow DOM、虚拟列表/懒加载、DOM 结构频繁改版、内部接口参数变化、用户登录态限制、长对话分页、推理过程混入正式回答、富文本/代码块/图片附件丢失。

因此技术路线不应只押注固定 DOM class。建议采用“平台插件优先 + 通用 DOM/Shadow DOM 兜底 + 可观测失败日志”的组合。
