# Memora — LLM 对话自动沉淀知识库 产品总结

> 一份初稿写到发布的全景回顾 | 2026-05-27

---

## 产品是什么

**Memora 是一款将大模型对话自动转化为结构化个人知识库的桌面工具。** 由浏览器扩展（数据抓取探头）和 Tauri 桌面应用（知识库主程序）两部分组成。用户与 ChatGPT、豆包、DeepSeek 等 LLM 的每一次深度对话，点击悬浮球即可一键抓取、AI 自动提炼为知识卡片，归入本地知识库，沉淀为属于你自己的"第二大脑"。

### 三大核心优势

**轻量私有，数据不出本机。** 基于 Tauri 2.0 构建，桌面应用包体积仅 ~10 MB（对比 Electron 方案的 ~150 MB）。所有对话数据和知识卡片存储在本地 SQLite 数据库中，API Key 由用户自行提供，不经过任何第三方服务器。你的知识只属于你。

**AI 驱动，从对话到卡片全自动。** 原始对话需要人工阅读整理才变成知识——这是传统笔记工具的痛点。Memora 在抓取对话后自动跑完一整套 4 步 AI Pipeline（清洗 → 切分 → 分类 → 生成），用户只需点一下悬浮球，30 秒内得到一张结构化知识卡片。单次 AI 调用成本约 ¥0.01，几乎可以忽略。

**9 平台覆盖，跨平台桌面客户端。** 浏览器扩展已适配豆包、元宝、DeepSeek、Kimi、Qwen、ChatGPT、Gemini 等主流 LLM 网页版。桌面客户端基于同一套 React 前端代码 + Rust 后端构建，同时产出 Windows 和 macOS 安装包，无需维护两套代码。

## 产品全链路：从点击悬浮球到知识卡片

### 1. 安装与连接

用户下载 Release 包（一个 zip），内含 `client/`（桌面客户端）和 `plugin/`（浏览器扩展）两个文件夹。在浏览器扩展管理页面导入插件后，扩展即通过 `localhost:17321` 与客户端建立本地 HTTP 连接，无需额外配置。

> **技术实现**：扩展为 Chrome Extension MV3 架构，manifest 声明 `http://localhost/*` 与 `http://127.0.0.1/*` 权限，通过 Background Service Worker 代理请求，绕过跨域限制向本地客户端发送数据。客户端侧由 Rust 的 axum 框架启动本地 HTTP Server 监听 17321 端口。

### 2. 抓取：悬浮球背后的平台适配

用户在 LLM 网页中完成一段对话后，点击页面右下角的悬浮球（魔法棒图标）。扩展的 Content Script 立即启动抓取流程。

**抓取策略并非暴力 DOM 遍历，而是分层降级方案：**

| 优先级 | 策略 | 说明 |
|--------|------|------|
| P1 | 平台内部 API | 如豆包 `/im/chain/single`，直接获取结构化 JSON，最稳定 |
| P2 | 语义化 DOM 选择器 | 如 ChatGPT `data-testid` 属性，平台级稳定标识 |
| P3 | Shadow DOM 深度遍历 | 兜底方案，处理 Web Component 封闭的 Shadow Root |
| P4 | UI 旁路（复制/编辑态） | 从对话的编辑态或复制按钮提取完整文本 |

每个平台实现统一的 `ChatProvider` 接口（`match()` 识别平台 + `extractCurrent()` 提取对话），新增平台只需实现两个方法，不影响现有逻辑。这种策略模式将平台差异隔离在各自的 Provider 文件中，确保了可扩展性。

> **技术实现**：扩展采用 TypeScript，Content Script 注入到 LLM 页面中运行。平台 Provider 接口定义 `id` / `match(url)` / `extractCurrent()` 三个核心方法，7 个已支持的平台各有独立实现文件。抓取结果为标准化的 `RawConversation` 结构（platform / conversationId / title / url / messages[] / capturedAt）。

### 3. 清洗：从原始 DOM 碎片到干净问答对

抓取下来的对话原始文本包含大量噪声：角色标签（"豆包："、"AI："）、系统消息（"以上是新的对话"）、时间戳前缀、重复连续的相同角色消息。这些噪声如果不处理，会直接污染后续 AI 提取的知识质量。

**Memora 实现了 4 步对话清洗管线：**

1. **normalize_role** — 将各种平台的角色标签统一为标准格式（User / Assistant）
2. **clean_content** — 按平台规则去除系统消息、时间戳、无关 UI 文本
3. **merge_consecutive** — 合并同一角色连续发送的多条消息（避免 LLM 输出被切碎）
4. **clean_conversation** — 过滤空消息、过短消息（如单独的"谢谢"）、纯表情等低信息量内容

> **技术实现**：清洗逻辑在前端 Demo 和 Rust 后端各有一份完全对等的实现。Demo 端为 JavaScript/Node.js，Tauri 端为 Rust 原生实现，两边跑相同的算法和正则规则，保障了"Demo 调试通过 = 客户端效果一致"的可靠性。清洗后的数据存入 `clean_conversations` 表，清洗前的原始数据保留在 `raw_conversations` 表，确保链路可追溯。

### 4. 话题切分：识别对话中的主题边界

用户的 LLM 对话往往跨越多个话题——前 20 轮在讨论数据库优化，后 15 轮转到了前端部署方案。如果不做切分，整段对话丢给 AI 总结，产出的卡片会主题混杂、可读性差。

**话题切分模块通过 LLM 分析对话轮次，将单一长对话拆分为多个独立的子话题块。** 核心原则是"合并优先"：同一主题下的子话题不拆分，只有在话题发生明显转换时才切分。同时设定了最小拆分阈值——至少 3 轮同一个话题才算一个块，连续 1-2 轮的跳跃会被合并到相邻主话题中。

> **技术实现**：切分器是一个独立的 system prompt（`docs/prompts/topic-split/`），以结构化 JSON 输出切分点数组。输入为清洗后的对话全文，输出为 `[{ start: 消息索引, end: 消息索引, summary: "子话题概括" }]`。这是 Pipeline 的第一次 LLM 调用。

### 5. 意图分类：10 类意图精准路由

切分好的每个话题块，需要判断它属于哪种对话类型——是在问一个概念、想让 AI 改代码、还是在头脑风暴？不同类型需要不同的信息提取策略。

**Memora 定义了 10 个意图大类，覆盖用户与 LLM 的全部使用场景：**

| 意图 | 特征信号 | 占比 |
|------|---------|------|
| 概念理解 | "是什么""为什么""原理" | 20% |
| 事实查询 | "谁""哪年""多少钱" | 15% |
| 操作指南 | "怎么""如何配置""报错了" | 15% |
| 内容创作 | "帮我写""生成""写代码" | 20% |
| 技能学习 | "教我""学习计划""备考" | 10% |
| 文本处理 | "润色""翻译""转成 JSON" | 10% |
| 规划决策 | "该选哪个""分析利弊" | 5% |
| 头脑风暴 | "帮我想想""来 10 个创意" | 5% |
| 交互陪伴 | "扮演""模拟""心情不好" | 3% |
| 其他 | 无法归类的边缘情况 | 2% |

> **技术实现**：分类器采用"轻量决策树 + few-shot 示例"结构。Prompt 从最初的 ~300 行压缩到 ~110 行，以 if-else 决策树格式呈现分类逻辑，在 `## 典型范例` 节中嵌入 13 个标注好的真实对话示例供 LLM 参考。分类器输出 card_type（中文值），同时 `intent_by_key()` 支持中英文双向映射作为兼容层。这是 Pipeline 的第二次 LLM 调用。

### 6. 卡片生成：每个意图有专属的"笔法"

分类完成后，对话数据被路由到对应意图的 Prompt 模板，进入**信息提取与卡片生成**阶段。这是整条 Pipeline 中最核心的一步。

**不同意图的"笔法"完全不同：**
- **概念理解**类卡片侧重认知突破和遗留疑问，输出 narrative（概念演变叙事）
- **操作指南**类侧重可执行步骤清单和常见坑点，输出 narrative + outputs（步骤清单）
- **内容创作**类对产出正文基本不压缩，保留完整代码/文案
- **事实查询**类压缩程度极高，只保留 1-2 行核心答案
- **头脑风暴**类侧重探索方向清单和创意草稿

每个意图的 Prompt 模板独立维护在 `docs/prompts/<intent>/` 目录下，包含：角色设定（该意图领域的 AI 角色）→ 逐字段 JSON 描述（每个输出字段的定义和示例）→ 叙事规则（narrative 的写作风格和字数要求）→ 多示例（带 `{{conversation}}` 占位符）。

> **技术实现**：Pipeline 会按意图加载对应的 Prompt 文件，通过 `{{conversation}}` 占位符替换注入对话文本，组装为完整的 system + user message 发送给 LLM。这是 Pipeline 的第三次 LLM 调用。AI 输出的 JSON 经过 `sanitizeContent()` 7 步正则质检（去除角色标签残留、空字段补位、HTML 标签剥离等）后写入数据库。

### 7. 去重：避免"同一段话存两遍"

用户可能在短时间内对同一段对话多次点击悬浮球，或者在话题切分时产生了与已有卡片高度重叠的新卡片。Memora 通过 **7 条综合去重规则** 在卡片入库前进行语义级去重：

- **card_type 相同** + **original_question 语义相似度**（Jaccard 相似度 > 阈值）
- **title 高度相似**（编辑距离或关键词重叠 > 阈值）
- **narrative 前 200 字重叠度** > 阈值

满足其中任意条件即视为重复。如果仅部分相似，则保留较新的卡片、归档旧的。

### 8. 最终知识卡片的数据字段

经过以上全链路处理后，每张知识卡片包含以下结构化字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 卡片唯一标识 |
| `title` | string | AI 生成的对话主题（可编辑） |
| `original_question` | string | 用户的原始核心问题 |
| `card_type` | string | 10 类意图之一，中文值（可下拉切换） |
| `narrative` | HTML | 核心叙事——AI 对这段对话的知识提炼（富文本） |
| `full_output` | JSON | AI 的完整原始输出 |
| `tags` | string[] | 层级标签（`父标签/子标签`），AI 生成 + 用户可编辑 |
| `unresolved_questions` | string[] | AI 识别出的遗留问题（富文本） |
| `source` | object | 来源平台、URL、对话 ID、抓取时间 |
| `starred` | boolean | 是否收藏 |
| `archived` | boolean | 是否归档 |

## 客户端：不只是看，而是管理和沉淀

在 Memora 客户端中，用户可以围绕知识卡片进行完整的管理操作：

**浏览与筛选** — 首页为 3 列便签墙布局（Tauri 窗口 1440×860），卡片以 230px 最小宽度自适应排列。左侧 260px 侧边栏提供按意图分类的导航和可点击筛选的标签云。顶部搜索栏支持基于 **SQLite FTS5 全文搜索引擎** 的关键词检索，搜索范围覆盖标题、narrative 和标签。

**收藏与删除** — 卡片列表每张卡片右上角有 ⋮ 更多菜单，支持一键收藏（⭐）和删除（带确认弹窗）。操作后页面通过局部状态覆写模式即时反映变化，无需等待后端刷新。

**导出** — 单张卡片支持导出为 TXT / Markdown / PDF / 图片四种格式。导出内容为用户可选择的范围（标题 + narrative + 标签 + 来源），原始对话不暴露。

**统计面板** — 展示知识总量、10 类意图分布饼图、平台来源分布、高频标签排行 Top 10。只做正向数据呈现，不做惩罚性统计。

**富文本编辑** — 详情页的 narrative 和 unresolved_questions 字段支持所见即所得编辑。基于 **TipTap**（ProseMirror 的 React 封装）实现，工具栏包含加粗、斜体、下划线、荧光笔高亮（2 色）等格式功能。编辑内容以 HTML 格式存储，渲染前经 **DOMPurify** 做 XSS 安全净化。800ms 防抖自动保存，细节质感完整。

> **技术实现**：`api.ts` 适配层支持 HTTP/Tauri IPC 双模式通信，同一套前端代码同时服务于 Demo 环境和 Tauri 客户端。5 个独立视图组件共存在 `App.tsx` 中（列表 ListView / 详情 CardDetail / 收藏 Starred / 统计 Statistics / 设置 Settings），通过 React state 切换而非路由。

## 长期价值：从对话流水到知识体系

把一次对话变成一张卡片只是起点。用户长期使用 Memora 后，会发生几件事：

**跨平台知识统一。** 原本散落在豆包、ChatGPT、DeepSeek 等 7 个平台上的对话，全部汇聚到同一处，按意图类型自动归类。你在豆包上问过的 Python 问题和在 ChatGPT 上讨论的架构方案，都出现在"编程"标签下。

**用碎片对话搭建领域框架。** 每次对话沉淀一张卡片，日积月累便形成按主题索引的知识网络。当你想回顾"异步编程"相关知识时，不用翻遍各平台的对话历史，直接在 Memora 中搜索标签就能找到所有相关卡片——包括当时 AI 给出的核心见解和你的原始问题。

**隐私可控的知识主权。** 所有数据以 SQLite 文件存储在本地（`%APPDATA%/com.memora.app/`），一个文件即可完整备份或迁移。不需要云服务、不必担心平台下架历史记录、也不用将个人对话上传给第三方。你的对话，你的知识，你说了算。

**AI 生成 + 人工策展的协作模式。** AI 负责第一轮提炼（省去大量手动整理时间），用户负责审阅和精修（编辑 narrative、修正分类、补充标签）。两种能力各展所长，最终沉淀下来的知识卡片，是人与 AI 协作完成的、真正属于你的"第二大脑"。

## 技术架构一瞥

| 层级 | 技术 | 核心职责 |
|------|------|---------|
| 浏览器扩展 | Chrome Extension MV3 + TypeScript | 9 平台悬浮球注入 + 分层抓取 + HTTP POST |
| 前端界面 | React + Vite + TypeScript + Tailwind CSS | 列表/详情/收藏/统计/设置 5 视图 + TipTap 富文本编辑 |
| 后端服务 | Rust + axum (HTTP Server) | HTTP API（11 条路由）、4 步 AI Pipeline、对话清洗 |
| 数据存储 | SQLite (sqlx) | 6 表（raw/clean/cards/settings/stats/fts）+ FTS5 全文搜索 |
| 桌面框架 | Tauri 2.0 | 窗口管理、系统托盘、appDataDir、跨平台构建 |
| AI 引擎 | OpenAI 兼容 API（用户自带 Key） | 3 次 LLM 调用：切分 → 分类 → 生成 |
| 桌面通信 | HTTP localhost:17321 | 扩展 ↔ 客户端的本地安全通信 |
| 跨平台 | Tauri build (Windows .msi + macOS .dmg) | 一份代码双平台输出 |

## 关键产品决策回顾

| 决策 | 选择 | 理由 |
|------|------|------|
| 桌面框架 | Tauri 2.0 | ~10MB 包体积、Rust 安全性、Tauri 自动抹平平台差异 |
| 数据存储 | SQLite | 文件级备份迁移、比 IndexedDB 可靠、FTS5 全文搜索 |
| AI 架构 | 4 步 Pipeline + 10 意图专属 Prompt | 单一通用 Prompt 质量无法满足不同对话类型的个性化提取需求 |
| 抓取策略 | 分层降级（内部 API → DOM → Shadow DOM） | 单一策略无法应对平台多样性和接口变更风险 |
| 内容范围 | 纯文本 + 代码 + 表格，图片只存 URL | 文本内容占知识价值的 95%+，base64 会使数据库膨胀 10-100 倍 |
| 隐私策略 | 全本地存储 + 用户自带 Key | 不持有任何用户数据或凭证，从根本上消除隐私泄露风险 |
| 去重策略 | 多维度语义相似度（7 规则） | 简单 hash 去重无法识别同一段对话的多轮交互变体 |
