# PRD v1：LLM 对话自动沉淀知识库工具

> 更新时间：2026-05-18
> 版本：v1（基于产品需求简报 + 技术可行性讨论整理）

---

## 1. 产品定位

**一句话：** 把你和大模型的对话，自动变成可检索的个人知识库。

**核心痛点：** 用户日常与 ChatGPT、豆包、DeepSeek 等大模型对话，产出大量知识、方案、代码和决策依据，但几乎不会手动整理；各平台历史记录杂乱、不可跨平台检索，有价值的内容持续沉没。

**核心价值：** 一键（或自动）将散落在各平台的对话提炼为结构化知识卡片，归入本地私有知识库，随时可检索、可复盘、可导出。

---

## 2. 目标用户

| 维度 | 描述 |
|------|------|
| 画像 | 高频使用 2+ 个网页版 LLM 的知识工作者 |
| 典型场景 | 程序员用 GPT 写代码、产品经理用豆包做方案、研究者用 DeepSeek 查文献 |
| 共性需求 | 把对话变笔记/方案库/代码片段库，且不想手动复制整理 |
| 隐私倾向 | 偏好本地存储，不想把对话再上传到第三方 |

---

## 3. 核心使用场景

### 场景 A：一键沉淀（MVP 核心）

```
用户在豆包/GPT 聊完一个话题
        ↓
点击浏览器页面的悬浮球
        ↓
扩展自动抓取完整对话（内部接口优先）
        ↓
发送到桌面应用后台
        ↓
AI 自动生成知识卡片（主题、疑惑、见解、产出、标签）
        ↓
归入本地知识库
        ↓
用户在桌面应用中搜索、浏览、导出
```

### 场景 B：智能提醒（Phase 2）

```
用户在 LLM 页面进行了一段对话后离开
        ↓
扩展检测到对话空闲 > N 分钟
        ↓
悬浮球变为提示状态（"有新对话可沉淀"）
        ↓
用户一键确认 → 执行抓取+总结+入库
```

### 场景 C：自动沉淀（Phase 3，可选）

```
扩展后台持续监听 LLM 页面活动
        ↓
按时间窗口自动分段（如 30 分钟无活动视为一段）
        ↓
自动抓取 + 总结 + 入库
        ↓
仅在通知栏显示结果
```

> **"对话结束"没有明确的技术信号，是核心设计难点。** 分阶段实现，从手动到自动，让用户始终有控制感。

## 4. 产品形态

### 4.1 整体架构：Tauri 桌面应用 + 浏览器扩展

```
┌─────────────────────────────────────────────────┐
│                  用户浏览器                       │
│  ┌──────────────┐   ┌────────────────────────┐   │
│  │  LLM 网页     │   │  浏览器扩展（轻量）      │   │
│  │  (豆包/GPT..) │   │  · 悬浮球               │   │
│  │              │   │  · 平台 Provider 抓取    │   │
│  └──────────────┘   └───────────┬────────────┘   │
│                                 │ HTTP POST       │
└─────────────────────────────────┼─────────────────┘
                                  │ localhost:17321
┌─────────────────────────────────┼─────────────────┐
│            Tauri 桌面应用        │                  │
│  ┌──────────────────┐  ┌───────▼──────────────┐   │
│  │  知识库主界面      │  │  后端服务 (Rust)       │   │
│  │  · 对话列表/搜索   │  │  · 接收扩展抓取数据     │   │
│  │  · 知识卡片详情    │  │  · 调 AI API 总结      │   │
│  │  · 标签筛选       │  │  · SQLite 持久化        │   │
│  │  · Markdown 导出  │  │  · 文件导入导出         │   │
│  │  · 设置/API Key   │  │  · 本地 HTTP 服务       │   │
│  └──────────────────┘  └───────────────────────┘   │
│  ┌──────────────────┐                              │
│  │  系统托盘         │  全局快捷键: Cmd/Ctrl+K     │
│  └──────────────────┘                              │
└────────────────────────────────────────────────────┘
```

### 4.2 为什么不用纯浏览器扩展？

| 问题 | 说明 |
|------|------|
| 知识库界面空间不足 | Side Panel/Popup 只有窄条，无法做复杂的浏览、筛选、详情阅读 |
| 无独立窗口 | 用户想"打开知识库认真翻一翻"时，必须先开浏览器再开侧边栏 |
| 跨浏览器数据隔离 | Chrome 和 Edge 数据不互通 |
| 没有全局快捷键 | 无法做到系统级随时唤出 |
| 存储不可靠 | IndexedDB 可能被浏览器清理 |
| 无法后台运行 | 浏览器关了就没有自动发现能力 |

### 4.3 为什么选 Tauri 而非 Electron？

| 对比 | Tauri 2.0 | Electron |
|------|-----------|----------|
| 包体积 | ~10MB | ~150MB |
| 内存占用 | 低 | 高 |
| 跨平台 | ✅ 同一份代码构建 Windows (.msi) + Mac (.dmg) | ✅ 同 |
| 平台差异处理 | Tauri API 自动抹平（路径、快捷键、托盘等） | 需自行处理 |
| 安全性 | Rust 后端，权限系统更严格 | Node.js 后端 |
| 移动端扩展 | Tauri 2.0 支持 iOS/Android | 不支持 |

### 4.4 跨平台策略

**一次编写，两端构建，无需维护两套代码：**

- 前端（React + HTML/CSS）：100% 平台无关
- 后端（Tauri Rust API 调用）：Tauri 自动处理平台差异
- 构建：`tauri build` 指定 target 即可产出 .msi（Windows）或 .dmg（Mac）
- 唯一需注意的 ~1% 差异：
  - 快捷键用 `CmdOrCtrl+K`（Tauri 自动映射）
  - 数据路径用 `appDataDir()` API（自动返回正确路径）
  - 窗口按钮位置 CSS 适配

## 5. 技术架构

### 5.1 浏览器扩展（抓取探头）

**职责单一：只负责识别平台 + 抓取对话 + 传给桌面应用。**

| 组件 | 职责 |
|------|------|
| Content Script | 注入悬浮球、识别当前平台 |
| Platform Provider | 按平台实现抓取逻辑（内部接口 / DOM / Shadow DOM） |
| Background Service Worker | 通过 HTTP POST 将抓取结果发送到桌面应用 |

### 5.2 平台 Provider 接口

```ts
interface ChatProvider {
  id: string;                                    // "doubao" | "chatgpt" | "deepseek" | ...
  match(url: string): boolean;                   // 匹配当前页面
  extractCurrent(): Promise<RawConversation>;    // 抓取当前对话
  fetchById?(id: string): Promise<RawConversation>; // 按 ID 拉取历史对话
  injectButton?(): void;                         // 注入悬浮球
}

interface RawConversation {
  platform: string;
  conversationId: string;
  title?: string;
  url: string;
  messages: Array<{
    role: "user" | "assistant";
    content: string;
    timestamp?: string;
  }>;
  capturedAt: string; // ISO datetime
}
```

### 5.3 抓取优先级

1. **平台内部接口读取**（最稳）— 如豆包 `/im/chain/single`
2. **平台稳定语义选择器** — 如 ChatGPT `data-testid` 属性
3. **通用 DOM / open Shadow DOM 深遍历** — 兜底方案
4. **复制按钮、编辑态、分享页** — UI 旁路
5. **OCR / 截图** — 极低优先级兜底

### 5.4 桌面应用架构

| 层 | 技术 | 职责 |
|----|------|------|
| 前端 | React + TypeScript | 知识库 UI、设置页、导入导出 |
| 后端 | Tauri (Rust) | 本地 HTTP 服务、SQLite、API 调用代理、文件系统、系统托盘 |
| 通信 | HTTP localhost:17321 | 扩展 → 桌面应用的本地通信 |

### 5.5 数据分层

```
Raw（原始抓取结果）→ Clean（清洗后纯文本问答）→ KnowledgeCard（AI 总结卡片）
```

### 5.6 知识卡片结构

```json
{
  "id": "uuid",
  "title": "对话主题",
  "original_question": "用户原始疑惑",
  "insights": ["学习收获与见解 1", "见解 2"],
  "outputs": ["可复用落地产出 1", "产出 2"],
  "tags": ["标签1", "标签2"],
  "source": {
    "platform": "doubao",
    "url": "https://www.doubao.com/chat/xxx",
    "conversation_id": "xxx",
    "captured_at": "2026-05-18T01:00:00Z"
  },
  "raw_messages": [...],
  "clean_messages": [...],
  "created_at": "2026-05-18T01:00:30Z",
  "updated_at": "2026-05-18T01:00:30Z"
}
```

### 5.7 本地存储

- **数据库：** SQLite（比 IndexedDB 更可靠，数据就是文件，可备份可迁移）
- **存储位置：** Tauri `appDataDir()` 自动确定
- **备份导出：** 支持导出为 JSON / SQLite 文件

## 6. AI 总结方案

### 6.1 任务性质

**整理总结是轻推理任务**，不需要强模型。本质是"结构化信息提取 + 格式化输出"：
- 输入一段对话 → 模型读懂内容
- 按固定模板输出 → 标题、疑惑、见解、产出、标签

### 6.2 推荐模型与成本

| 模型 | 输入/1M tokens | 输出/1M tokens | 单次成本（~10K in + 1K out） | 推荐度 |
|------|---------------|---------------|---------------------------|--------|
| **GPT-4.1 nano** | $0.10 | $0.40 | ≈ ¥0.01 | ⭐⭐⭐ MVP 首选 |
| **Gemini 2.5 Flash-Lite** | $0.10 | $0.40 | ≈ ¥0.01 | ⭐⭐⭐ 同级别 |
| **DeepSeek V3** | ¥2 | ¥8 | ≈ ¥0.03 | ⭐⭐ 国内首选 |
| GPT-4.1 mini | $0.40 | $1.60 | ≈ ¥0.04 | ⭐ 质量更好时用 |
| Gemini 2.5 Flash | $0.30 | $2.50 | ≈ ¥0.05 | ⭐ 同上 |

### 6.3 月度成本估算（GPT-4.1 nano）

| 使用强度 | 次数/月 | 月成本 |
|---------|--------|--------|
| 轻度 | 150 | ≈ ¥0.27 |
| 中度 | 450 | ≈ ¥0.81 |
| 重度 | 900 | ≈ ¥1.62 |
| 重度 + 长对话 (30K input) | 900 | ≈ ¥8.10 |

**结论：API 成本极低，不是商业模式的障碍。**

### 6.4 长对话优化策略

- **截取策略**：大多数对话关键信息集中在后半段，可只发送最近 N 轮
- **分片总结**：长对话分段总结再合并，两次小调用比一次大调用更便宜
- **渐进式总结**：追加问题时只总结新增部分，与已有卡片合并
- **缓存命中**：相同 system prompt 部分走 DeepSeek 缓存，输入成本降 75%

### 6.5 本地模型（可选，零成本）

- 用 Ollama 跑 Qwen2.5-7B / Llama3-8B
- 完全免费，但需用户有 8GB+ 显存
- 作为"离线模式"选项，MVP 不做默认

## 7. MVP 范围（Phase 1）

### 7.1 必做

| 功能 | 说明 |
|------|------|
| 豆包单平台抓取 | 通过内部接口 `/im/chain/single` 分页拉取 |
| 浏览器悬浮球 | 一键触发抓取 |
| Tauri 桌面应用 | 知识库主界面 + 后台服务 |
| 扩展↔桌面通信 | HTTP localhost:17321 |
| AI 总结 | GPT-4.1 nano / DeepSeek V3，生成知识卡片 |
| SQLite 存储 | raw + clean + card 三层存储 |
| 知识库列表 | 按时间倒序，显示标题、平台、标签 |
| 知识卡片详情 | 完整展示主题、疑惑、见解、产出、原始对话 |
| 关键词搜索 | 标题 + 内容全文搜索 |
| 标签筛选 | 点击标签过滤 |
| Markdown 导出 | 单条卡片导出为 .md 文件 |
| API Key 管理 | 用户自带 Key，设置页填入 |
| 系统托盘 | 最小化到托盘，后台运行 |

### 7.2 不做（MVP）

| 排除项 | 原因 |
|--------|------|
| 多平台支持 | 先验证豆包单平台跑通 |
| 自动发现/智能提醒 | Phase 2 再做 |
| 云同步 | MVP 先本地 |
| 团队协作 | 个人工具定位 |
| 富文本/图片抓取 | 先支持纯文本和代码块 |
| OCR / 截图 | 极低优先级 |
| 付费/订阅系统 | 先免费 + 自带 Key |

## 10. 最大风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 平台内部接口变更/封禁 | 抓取失败 | 多层兜底策略（接口 → DOM → Shadow DOM），失败日志可观测 |
| "对话结束"判定模糊 | 自动场景体验差 | 分阶段：手动 → 提醒 → 自动，用户始终有控制权 |
| 用户不愿装两个东西（扩展+桌面应用） | 获客门槛高 | 安装引导要丝滑；扩展极轻量，桌面应用包体积小（~10MB） |
| AI 总结质量不稳定 | 知识卡片不好用 | 小模型对固定模板任务足够稳定；可升级模型 |
| 跨平台 UI 一致性 | 维护成本 | Tauri 自动抹平；Web 渲染天然一致 |
